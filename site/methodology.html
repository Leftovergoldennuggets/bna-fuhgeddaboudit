Methodology

This page documents our complete data processing methodology, including limitations, caveats, and known biases. Transparency is central to this project — we want you to understand exactly how we produced every number on this website.

Data Sources

NHTSA Standing General Order (SGO) Data

The National Highway Traffic Safety Administration requires all companies operating Level 3+ autonomous vehicles to report any crash within a set timeframe. This creates the most comprehensive public record of AV incidents in the United States.

NHTSA provides two separate files:

  • Pre-June 16, 2025 (Amendment 2): The archived dataset covering reports filed before NHTSA updated their reporting format.
  • Post-June 16, 2025 (Amendment 3): The current active dataset with an updated column structure.

These files contain crashes from all AV companies (Waymo, Cruise, Zoox, Tesla FSD, etc.). We filter for Waymo-only crashes using the "Reporting Entity" field.

Waymo Safety Impact Data Hub

Waymo publishes a curated dataset at their Safety Impact Data Hub. This dataset (CSV2) includes crash type classifications, outcome group membership, and links to the corresponding SGO Report IDs.

The Hub data is updated approximately quarterly and currently covers September 2020 through September 2025.

Data Processing Pipeline

Our data pipeline runs in five automated steps:

Step 1: Download

We download all three raw CSV files directly from NHTSA and Waymo's public servers. The pipeline includes error detection for cases where URLs may have changed (particularly the Waymo Hub URL, which includes a date range in the filename that changes quarterly).

Step 2: Merge and Clean

This is the core data processing step:

  1. Filter: From the NHTSA data, we keep only rows where Reporting Entity = "Waymo LLC". The full NHTSA dataset includes many other companies.
  2. Harmonize: The pre- and post-June 2025 files have different column names (due to the Amendment 2 → 3 transition). We rename columns from the older format to match the newer format.
  3. Combine: The two NHTSA files are stacked into a single dataset.
  4. Deduplicate: Some crashes have multiple report versions (updates filed over time). We keep only the latest version of each report, identified by the Report ID and Report Version fields.
  5. Merge: We perform a left join of the Waymo Hub data with the NHTSA data, using the Hub's "SGO Report ID" column and NHTSA's "Report ID" column. The Hub is the base — all Hub rows are preserved, and NHTSA fields are added where a match exists.

Step 3: Compute Statistics

Every number displayed on the website is computed in this step and saved to a JSON file. This includes:

  • Total crash counts and city breakdowns
  • Time-of-day analysis (rush hour, late night, daytime/nighttime splits)
  • Crash type distributions
  • Severity indicators (police-reported, injury-reported, airbag deployment, serious injury)
  • Location type analysis (intersection, street, parking, highway)
  • Day-of-week patterns

Step 4: Generate Map Data

Each crash record is geocoded and converted into a simplified JSON object for the Leaflet map. We use a multi-strategy geocoding approach with OpenStreetMap's Nominatim service:

  1. Intersection parsing: Addresses like "7th Street & Market Street" are split and geocoded as intersections.
  2. Street address lookup: Full addresses are geocoded directly.
  3. Fallback: If geocoding fails, the crash is placed at the city center and flagged as an approximate location.

Results are cached locally to avoid repeated API calls. The current geocoding success rate is 92% (1,026 of 1,119 mapped crashes placed at their real street location). Each record also includes severity flags (has_injury, is_serious, is_vulnerable_road_user) used for color-coding markers on the map.

Step 5: Extract Serious Incidents

Crashes involving moderate, serious, or fatal injuries are extracted into a separate file with their full NHTSA narratives, enabling the detailed incident view on the map.

Limitations and Caveats

Crash Locations and Geocoding

NHTSA redacts GPS coordinates from public data, but provides street addresses for nearly all crashes in the "Location Address / Description" field. We geocode these addresses using OpenStreetMap's Nominatim service with a multi-strategy approach (intersection parsing, direct address lookup, city-bounded search).

Current accuracy: 92% of crashes (1,026 of 1,119) are placed at their real street location. The remaining 8% fall back to city-center placement and are flagged as approximate. Accuracy varies by city:

  • San Francisco: 98%, Los Angeles: 98%, Austin: 97%
  • Atlanta: 85%, Phoenix: 77% (some reports have less detailed addresses)

Geocode results are cached in geocode_cache.json to ensure reproducibility and avoid repeated API calls.

Reporting Thresholds

The NHTSA SGO requires reporting of any crash, but companies may interpret "crash" slightly differently. Additionally, some minor incidents (e.g., a parked car being bumped with no damage) might not be consistently reported across all companies.

Severity Assessment

We report severity using four independent boolean indicators rather than a composite score:

  • Police-reported: Whether law enforcement responded to the scene
  • Injury reported: Whether any injury (of any severity) was reported
  • Airbag deployed: Whether any vehicle's airbag deployed
  • Serious injury: Whether a suspected serious injury was reported

We deliberately avoid creating composite "severity scores" because the weighting of different factors would be arbitrary and potentially misleading.

Waymo Safety Comparisons

The statistics comparing Waymo to human drivers (e.g., "90% fewer serious crashes") come from Waymo's own peer-reviewed research paper published in the journal Traffic Injury Prevention. While the paper has undergone peer review, it is important to note:

  • The study was authored by Waymo researchers
  • The comparison benchmarks may not perfectly match Waymo's operating conditions (urban environments, specific cities, time of day, weather)
  • Autonomous vehicles currently operate in a limited set of conditions that may be systematically different from the average human driving environment

Dataset Coverage

The Waymo Hub dataset does not include every NHTSA-reported crash. Some NHTSA crashes are excluded by Waymo's own methodology (e.g., incidents that don't meet their definition of a reportable crash). Our merge process preserves all Hub crashes and flags NHTSA-only crashes separately.

Temporal Limitations

Crash time data comes from the "Incident Time" field in the NHTSA reports. A small number of records have missing or unparseable times, which are excluded from time-of-day analyses. The overall coverage is high (typically >99% of records have valid time data).

Automated Updates

This website is designed to be "evergreen" — every statistic is computed from data, and the entire pipeline can be re-run when new data becomes available.

A GitHub Actions workflow is configured to:

  1. Download the latest data from NHTSA and Waymo
  2. Run the full processing pipeline
  3. Regenerate all JSON data files
  4. Commit and push any changes to the repository

This ensures the website stays current without manual intervention. However, the Waymo Hub URL changes quarterly (the filename includes a date range), so this URL may need periodic manual updates in the configuration file.

Reproducibility

To reproduce our analysis:

  1. Clone the repository
  2. Install dependencies: pip install -r requirements.txt
  3. Run the pipeline: make data
  4. View the results: make serve

Geocoding results are cached in geocode_cache.json (committed to the repository), so map layouts are identical across runs without re-querying the geocoding service.
